# FAQ

## Why are booleans encoded as ±1?

Boolearn uses the convention `True/1 → +1` and `False/0 → −1`. This makes dot-products and “sign” thresholding natural in the BTF formula:

$y = \mathrm{sgn}(w\cdot x)$

See [CONCEPTS.md](CONCEPTS.md) for the full setup.

## Why does my network have one extra input node?

In `.net` files generated by `layered`, node `0` is a **bias** input. It’s a constant input (always `False/0` in the original guide), and BTFs learn bias via the corresponding weight.

This is why a “4 input” problem can show up as `innodes = 5` in the `.net` header.

## I’m getting different solutions between runs — is that expected?

Yes. Two things contribute:

1. **Random initialization**: `train` seeds the RNG from the current time (`srand(time(NULL))`), so each run starts differently.
2. **Symmetries**: layered boolnets have symmetries (permutations within layers, and sign-flip/self-duality symmetries on hidden BTFs). So even if the function learned is “the same”, the weights can look different.

## What files does `train` write?

Given an `id` argument (the last argument to `train`), you’ll typically see:

- `<id>.cmd` — the command line (easy copy/paste)
- `<id>.gap` — progress / monitoring output
- `<id>.run` — multi-run summary if `trials > 1`
- `<id>.sol` — the learned weights / solution
- `<id>.gen` — only if you are in a generated-data mode (no input data)

## How do I watch training progress live?

The easiest is:

```bash
tail -f <id>.gap
```

(Or open it in a separate terminal and refresh.)

## What do the core hyperparameters mean?

From the original multiplier tutorial:

- `btfn` — BTF-norm parameter (used to set margins)
- `beta` — time-step value for the RRR constraint satisfaction algorithm
- `gam` — metric auto-tuning parameter
- `epochs` — number of lines written to the gap file (log-spaced in iteration count)
- `itermax` — maximum number of iterations
- `gapstop` — termination threshold on gap
- `trials` — repeat runs for statistics

See [EXAMPLES.md](EXAMPLES.md) for the full walkthrough.

## What does `<<` mean in the docs?

In the original guide, the notation:

- ``param << value``

is used informally to mean “set `param` to `value`” in an example command.

It’s **not** C/bitshift syntax in this context — just a compact way to list suggested settings.
